## Grafeo - Fast, embeddable graph database built in Rust

[Grafeo on HN](https://news.ycombinator.com/item?id=43033457) | [grafeo.dev](https://grafeo.dev/)

A new graph database written in Rust hit the front page with support for multiple query languages (GQL, Cypher, Gremlin, GraphQL, SPARQL) and dual data models (Property Graph + RDF). What makes this interesting is the embeddable-first design—you can bundle it into Python/Node/Go/C#/Dart apps with zero external dependencies, or run it as a standalone server. Performance claims look good on LDBC benchmarks, and ACID transactions with MVCC isolation mean it's not just a toy.

## Tinybox ships $12K-$65K deep learning computers with 4x RTX GPUs

[tinygrad.org](https://tinygrad.org/)

George Hotz's tinygrad project now has hardware in stock—red boxes at $12K and green boxes with 4x RTX Pro 6000 Blackwell GPUs at $65K. The "exa" model is listed for 2027 at ~$10M. This is real shipping product, not vaporware, targeting researchers and small teams who want dedicated ML compute without cloud lock-in. The fact that these are actually available is the news—most custom ML hardware stays in preorder hell forever.

## The three pillars of JavaScript bloat

[HN discussion](https://news.ycombinator.com/item?id=43033820)

Top post on HN today dissecting why JavaScript bundles keep ballooning despite better tooling. The analysis covers dependency sprawl (every package imports 47 transitive deps), polyfill bloat (shipping code for browsers that died in 2018), and framework overhead (React's runtime alone is heavier than entire apps used to be). This isn't new ground, but it's a good catalog of what specifically makes modern web apps slow and why "just use Webpack" didn't fix it.

## Claude's 1M context window now GA with no long-context premium

[Simon Willison](https://simonwillison.net/2026/Mar/13/1m-context/)

Anthropic made the full 1M token context window available for Opus 4.6 and Sonnet 4.6 at standard pricing across the entire window. OpenAI and Gemini both charge extra once you cross 200K-270K tokens, so this is a real cost advantage for long-document work. If you're summarizing giant codebases or legal docs, Claude just became cheaper to run at scale.

## Give Django your time and money, not your tokens

[Tim Schilling](https://www.better-simple.com/django/2026/03/16/give-django-your-time-and-money/)

Blunt take on LLM-generated Django contributions: if you don't understand the ticket, the code, or the PR feedback, you're making the project worse. Reviewers can tell when they're talking to a facade instead of a human, and it kills the communal part of open source. Use AI as a tool if you want, but slapping GPT output into a PR without comprehension is just noise. This applies way beyond Django—every major OSS project is dealing with this now.

## Running 397B parameter Qwen on a 48GB MacBook using LLM in a Flash

[Simon Willison](https://simonwillison.net/2026/Mar/18/llm-in-a-flash/)

Dan Woods got Qwen3.5-397B-A17B running at 5.5+ tokens/sec on a 48GB M3 Max by implementing Apple's "LLM in a Flash" paper, which streams MoE expert weights from SSD instead of holding them all in RAM. The model is 209GB on disk (120GB quantized), but only the active experts per token load into memory. Claude Code helped run 90 experiments to optimize the Metal/Objective-C implementation. It's not clear how much quality degraded, but the fact that a nearly 400B model runs at all on a laptop is wild.
